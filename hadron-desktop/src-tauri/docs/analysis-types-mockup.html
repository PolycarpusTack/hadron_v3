Technical Root Cause
For developers
Affected Method
BMProgramSegmentDurations>>calculateTotalDuration

The BMProgramSegmentDurations>>calculateTotalDuration method attempted to send #asSeconds to a nil receiver. The nil originated from PSITxBlock>>endTime returning nil when the transmission block lacks a scheduled end time. This occurs when importing partial schedule data from the traffic system where end times are calculated rather than provided.

Trigger Condition
PSITxBlock with nil endTime passed to duration calculator during EPG export
💬
Plain English Explanation
For stakeholders

The system crashed while trying to calculate how long a TV program segment lasts. This happened because the schedule entry was missing its end time - the system expected the traffic system to provide complete timing information, but received only a start time.

Business Impact: The 6PM EPG publication for BBC One failed. Downstream systems (automation, EPG distributors) did not receive the updated schedule. Manual intervention is required to republish.

Suggested Fixes
Prioritized remediation steps
Add nil guard in duration calculator
P0 ~2 hours

Add defensive nil check before calculating duration. Return zero or raise domain-specific exception.

Before
calculateTotalDuration
    | total |
    total := 0.
    segments do: [:seg |
        total := total + seg endTime asSeconds - seg startTime asSeconds].
    ^total
After
calculateTotalDuration
    | total |
    total := 0.
    segments do: [:seg |
        seg endTime ifNil: [
            WOnScheduleWarning signal: 'Segment missing end time: ', seg id printString.
            ^nil].
        total := total + seg endTime asSeconds - seg startTime asSeconds].
    ^total
Validate schedule entries on import
P1 ~1 day

Add validation step in traffic import to flag entries with incomplete timing data before they enter the scheduling workflow. Show warning dialog to user.

Implement automatic end time calculation
P2 ~1 week

When end time is missing, automatically calculate from start time + program duration from the media asset metadata. Requires integration with MAM system.

Why This Fix Works

The P0 fix adds defensive programming - checking for nil before using the value. This is a Smalltalk best practice when dealing with data from external systems.

Instead of crashing, the code now:

  • Detects the missing end time
  • Logs a warning for investigation
  • Returns nil to signal incomplete data (caller can handle gracefully)

The P1 fix addresses the root source - preventing bad data from entering the system in the first place. This is better long-term but requires more effort.
