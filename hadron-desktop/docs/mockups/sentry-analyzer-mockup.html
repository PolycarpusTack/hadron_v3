🚨 Detected Pattern: PostgreSQL Deadlock
Two concurrent Celery tasks are competing for locks on the program_segments and broadcast_slots tables. Task A holds a lock on program_segments row 1842 and waits for broadcast_slots, while Task B holds broadcast_slots and waits for program_segments row 1842.
🔍 Root Cause
The process_segment_batch() function in scheduler/handlers.py:142 acquires row-level locks in an inconsistent order. When processing batch updates, segments and their associated broadcast slots are locked in the order they appear in the batch, rather than in a deterministic order (e.g., by primary key).

When two concurrent Celery workers process overlapping batches, they can acquire locks in opposite orders, creating a classic deadlock cycle. The PostgreSQL deadlock detector aborts one transaction after the 5-second deadlock_timeout.

The breadcrumbs confirm: a BEGIN TRANSACTION followed by segment update, then slot update, then a blocked second segment update from a concurrent task.

🛠 Suggested Fixes
  • 1. Sort lock acquisition order. In process_segment_batch(), sort segment IDs before processing. Add segments.sort(key=lambda s: s.id) before the update loop. This ensures all tasks acquire locks in the same order.
  • 2. Use SELECT ... FOR UPDATE with NOWAIT. Replace the implicit locks with explicit SELECT ... FROM program_segments WHERE id IN (...) ORDER BY id FOR UPDATE at the start of the transaction. This acquires all needed locks upfront in a consistent order.
  • 3. Add retry with exponential backoff. Wrap the batch operation in a retry decorator: @celery_app.task(autoretry_for=(OperationalError,), retry_backoff=True, max_retries=3). This handles transient deadlocks gracefully while fix #1 or #2 are deployed.
