Template Authoring Guide

How Templates Work

Templates define what Hadron generates and how the AI formats it. A template is a JSON document with three layers:

1. Structure

Sections, ordering, and which ticket types map to which headings.

2. AI Instructions

Tone, word limits, and formatting rules that constrain the AI output.

3. Enrichment

Cross-references, screenshot suggestions, and metadata annotations.

Creating a Template

1.
Define sections — Map JIRA ticket types to output headings. For example, "filter": {"type": ["Story"]} captures all Story tickets into that section. You can also filter by label, component, or priority.
2.
Set the tone — The tone.instructions field is passed directly to the AI as a system prompt. Be specific: "Write for C-level executives. No jargon. Max 8 words per bullet." works better than "Be professional."
3.
Write the markdown template — Use {{#each}} to loop, {{#if}} for conditionals, and {variable} for interpolation. The AI fills {ticket.ai_description} at generation time — everything else is literal.
4.
Configure enrichment — Enable cross_references to have the AI trace linked issues and prior versions. Enable screenshots to have it detect UI-visible changes and insert placement hints.
5.
Set filters — Exclude ticket labels (internal, security), types (Sub-task), or statuses. Only tickets passing all filters are fed to the AI.

Cross-References

When enabled, the AI traces three sources to find connections to earlier releases:

Linked Issues

JIRA "is caused by", "relates to", "blocks" links pointing to tickets in older fixVersions.

Epic History

If a ticket belongs to an epic that spans versions, the AI references when the epic work began.

Shared Components

Tickets touching the same JIRA component as older tickets trigger an inferred relationship.

References appear as inline phrases like "Building on the JIRA integration introduced in v3.8.0" when inline: true, or as a separate "Related Versions" section when inline: false.

Screenshot Suggestions

The AI detects which tickets involve UI-visible changes using three signals:

UI Labels

Tickets with labels like ui, frontend, ux, or design.

Component Tags

JIRA components mapped as visual (e.g., "Dashboard", "Settings Panel", "Analysis View").

Attachment Types

Tickets with .png/.jpg/.gif attachments (mockups, designs) automatically flag.

Each suggestion includes an AI-generated caption describing what to capture. Placeholders use <!-- 📸 SCREENSHOT: ... --> in the markdown which renders as visual hints in preview mode.

Import & Export

Templates are portable JSON files. Share them across teams or version-control them alongside your project.

Export

Click the export icon on any template card, or use the "Copy" button on the JSON Schema preview in the editor. The exported .json file contains everything needed to reproduce the template.

Import

Click "Import .json" and select a template file. Hadron validates the schema, shows a preview, and lets you rename before saving. Duplicates are detected by template name.

Tips for Effective Templates

  • ‣ Be brutally specific in tone instructions. "No sentences over 15 words" beats "Be concise."
  • ‣ Use {ticket.ai_description} instead of {ticket.summary} — the AI rewrites the JIRA summary to match your tone.
  • ‣ Set word limits per section, not just globally. Features deserve more words than bug fixes.
  • ‣ Cross-references work best when your JIRA has consistent epic/linking hygiene. Garbage in, garbage out.
  • ‣ For screenshot suggestions, tag your JIRA components as visual/backend in Settings > Integrations. The AI uses this mapping.
  • ‣ Test templates against a past version first — generate notes for v3.8.0 and compare to what you manually wrote.
